Your current architecture looks like a solid foundation for a classic **3-Tier / Clean Architecture** pattern. Mapping your Presenter (Controllers), Domain (Services), and Data (Repositories) clearly separates responsibilities, which is exactly what you want when building a scalable enterprise application.

However, when introducing **localization (l10n) / internationalization (i10n)** to the backend, there are a few architectural bottlenecks in your current design—specifically regarding how the language state is handled and where translation logic actually belongs.

Here is an analysis of your current diagram, followed by how to handle backend localization effectively.

---

## 1. Analysis of Your Current Diagram

### The Good

* **Clear Responsibility Separation:** Your mapping from `Controller -> Service -> Repository` flows sequentially, which aligns perfectly with ASP.NET Core web API conventions.
* **Granular Domains:** Breaking things down into `機械 (Machine)`, `機械データ (Machine Data)`, and `ユーザー (User)` keeps the context clean.

### Areas for Improvement

* **The "Shared Resource Service" (`共有リソースService`) Bottleneck:** In your diagram, you have horizontal dependencies where your domain services depend on `共有リソースService` for translation (`翻訳機能`). If every service manually calls a translation method inside its business logic, your domain layer becomes cluttered with cross-cutting concerns.
* **Missing Language Context Flow:** There is no indication of how the backend *knows* which language the user wants. In a web API, this should be treated as request-scoped metadata, not something passed manually down into every single method signature.

---

## 2. Where Does the Language Live on the Backend?

In an ASP.NET Core backend, the requested language should **never be hardcoded or managed deep inside the business logic (Domain Layer)**. Instead, it should be captured at the entry point of the application and made available via dependency injection as context.

### The Entry Point: Middleware & `Accept-Language`

When your Angular frontend (`ngx-translate`) makes an API call, it should automatically attach an HTTP `Accept-Language` header (e.g., `Accept-Language: ja-JP` or `en-US`).

ASP.NET Core has a built-in middleware precisely for this: **`RequestLocalizationMiddleware`**.

* The middleware intercepts the request.
* It parses the header and automatically sets `CultureInfo.CurrentCulture` and `CultureInfo.CurrentUICulture` for the **current asynchronous execution thread**.

### Where Do the Error/Exception Translations Live?

Because the thread's culture is set automatically by the middleware, your Domain services do not need a `- 翻訳機能` (translation function) inside them. Instead, you use ASP.NET Core’s built-in **`IStringLocalizer<T>`**.

Here is how the clean architecture handles it without polluting the domain:

```
[ HTTP Request: Accept-Language: ja-JP ]
                 │
                 ▼
    [ Localization Middleware ] ──► Sets Thread.CurrentUICulture = "ja-JP"
                 │
                 ▼
       [ 機械Controller ]
                 │
                 ▼
        [ 機械Service ] ──► Encounter Validation Error? 
                 │          Throws exception or returns a translation key.
                 ▼
  [ Global Exception Filter ] ──► Uses IStringLocalizer to convert key to "ja-JP" text.
                 │
                 ▼
[ HTTP Response: 400 Bad Request (Localized Message) ]

```

---

## 3. Recommended Architectural Adjustments

To refine your design, here is how you should restructure the localization responsibilities across your tiers:

### A. Presenter / API Layer (The Validation & Exception Gate)

* **What stays here:** This is where raw localization strings are transformed into HTTP responses.
* **Validation Errors:** If you use `FluentValidation` or Data Annotations in your DTOs, hook them up to `IStringLocalizer`. The validation errors will be localized before they even hit your controllers.
* **Global Exception Handler:** Instead of catching exceptions and localizing them inside each Service method, catch them globally using an ASP.NET Core Middleware or an `IExceptionFilter`. The global handler injects `IStringLocalizer` and translates the exception key based on the current thread's culture.

### B. Domain / Service Layer (Pure Business Logic)

* **Remove `翻訳機能` from individual services.** Your services shouldn't care *how* a string is translated into Japanese or English; they should only care about business rules.
* If a service needs to throw an error, it should throw a custom exception with a **static error key** (e.g., `throw new MachineNotFoundException("ERROR_MACHINE_NOT_FOUND");`).
* Let the Presenter tier convert `"ERROR_MACHINE_NOT_FOUND"` into *"機械が見つかりません"* or *"Machine not found"*.

### C. Data / Repository Layer (Database Localization)

If you need to localize **dynamic data stored in the database** (e.g., if a machine's status or description has both a Japanese name and an English name stored in the DB), the structure changes:

* **Lookup Tables:** You can use a translation table pattern in Entity Framework (e.g., a `MachineDescription` table with columns `MachineId`, `LanguageCode`, and `Text`).
* Your Repositories can read `CultureInfo.CurrentUICulture.Name` to automatically append a `.Where(x => x.LanguageCode == currentCulture)` clause to your EF Core queries.

---

## 4. Frontend (`ngx-translate`) Integration Tip

Since you are using Angular with `ngx-translate`, keep your backend error handling elegant by matching keys:

1. When the backend returns a validation or application error, return a structured JSON object:
```json
{
  "errorCode": "MACHINE_UNDER_MAINTENANCE",
  "parameters": { "machineId": "M001" }
}

```


2. In your Angular app's `en.json` and `ja.json` files, map that exact key:
* **en.json:** `"MACHINE_UNDER_MAINTENANCE": "Machine {{machineId}} is currently under maintenance."`
* **ja.json:** `"MACHINE_UNDER_MAINTENANCE": "機械 {{machineId}} は現在メンテナンス中です。"`


3. Pass the backend's `errorCode` directly into Angular's `translate` pipe. This offloads the translation rendering completely to the client side for standard application flow, leaving the backend localizer strictly responsible for infrastructure/system-level exceptions.

### **Request Body (リクエストボディ)**

The JSON payload sent over the HTTP POST/PUT request to your ASP.NET Core backend API. Your Angular service takes the raw data from the frontend `ViewModel`, maps it to match your backend DTO schema, and passes it into the HTTP body.

Let’s break this down. First, I completely understand that you are locked into this specific requirement. In the real world, enterprise requirements don't always align with textbook security standards, and as an engineer, your job is to build the machine requested while making it as safe as possible within those boundaries.

Before writing the event flow, we need to address a critical technical detail regarding how cryptography works, because a direct **"decrypt the hashed database password"** step is mathematically impossible with standard hashing.

### A Quick Reality Check on "Hashing" vs. "Encryption"

* **Hashes are a one-way street:** Algorithms like BCrypt, Argon2, or SHA-256 cannot be "decrypted." Once you hash a password, you can never turn it back into plain text.
* **The Solution for Your Requirement:** To make your required workflow function exactly as you described (where the backend must decrypt things to compare them), the frontend and the database must use **reversible encryption** (like AES-256) rather than true hashing, or the frontend encrypts/hashes while the backend uses a verifiable hashing mechanism.

To prevent an attacker from simply copy-pasting a stolen frontend payload string into the login form and successfully logging in (a classic Pass-the-Hash/Replay attack), we can use a **timestamped payload or a unique initialization vector (IV)**.

## 3. Review of this Architecture

Since you mentioned you *have* to follow this design due to external requirements, here is how you can make it as resilient as possible under those constraints:

### Use AES for Frontend "Hashing"

Since the backend *must* decrypt the incoming string to get the original password text, the frontend cannot use a pure SHA-256 hash. It must use an encryption method like **AES**. You can hardcode a client-side obfuscated key/IV, or grab a dynamic public key from the backend when the login page loads (similar to an asymmetric RSA implementation).

### Preventing the Copy-Paste (Replay Attack)

You specified that copy-pasting the payload string shouldn't work. The best way to enforce this in your step 4.1 backend check is to pass a **timestamp** inside the frontend's encrypted string.

* If a malicious user inspects the network, copies the encrypted blob, and tries to send it 5 minutes later, the backend will decrypt it, see that the timestamp inside the payload is expired, and reject it—even though the password string matches.

This gives you a strong defense layer despite the unusual constraint! Does this approach to handling the frontend string decryption match up well with how you envisioned implementing the payload security?

## 2. Architectural Recommendation for Handling 6–7 Columns

Since you have 6 to 7 columns, saving them instantly can be tackled in two different ways depending on how you want your payload to look:

### Approach A: Send Single Rows (Highly Recommended for Instant Save)

Every time a checkbox is clicked, send *only* that specific column's change to the backend.

* **Payload:** `{ "userId": "U001", "columnName": "MachineStatus", "displayFlag": false }`
* **Pros:** Very lightweight payload, easy to map to your current `displaysetting table` schema.

### Approach B: Send the Entire Payload (Bulk Upsert)

Every time a checkbox is clicked, send the status of *all* 6–7 columns at once as an array.

* **Payload:** ```json
[
{ "columnName": "MachineID", "displayFlag": true },
{ "columnName": "MachineName", "displayFlag": true },
{ "columnName": "Status", "displayFlag": false }
]
```

```


* **Pros:** Backend code doesn't need to know *which* checkbox changed; it just wipes the old settings for that `userId` in the DB and inserts the new list, or loops and updates them. It guarantees frontend and backend are always perfectly in sync.

### 💡 Crucial Frontend Tip: Debouncing (Optimizing the UI)

Since this is an instant save feature, if a user quickly clicks 3 different checkboxes in one second, it will trigger 3 simultaneous HTTP requests to your ASP.NET backend. This can cause a "race condition" where the `lastupdate` timestamp in the database gets locked or overwritten incorrectly.

In Angular, you can use RxJS operators like **`debounceTime(300)`** or simply disable the checkboxes during the `ViewModel.isLoading` state (as mentioned in Step 3 and 6.1) to ensure the previous database write finishes before the user can trigger the next one!
