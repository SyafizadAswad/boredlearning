To answer your question: **Yes, you should handle exception localization on the backend** if those error messages are user-facing or need to be logged in a specific language. While you *could* just return error codes (like `ERROR_MACHINE_OFFLINE`) and let Angular's `ngx-translate` map them, standardizing localization in the backend ensures consistency across different potential clients (web, mobile, or third-party APIs).

In an n-tier .NET application with custom Middleware and Exception classes, the idiomatic way to handle this is using **`IStringLocalizer`** alongside your Centralized Exception Handling Middleware.

Here is exactly what you need to add and how to wire it up.

---

## 1. The Core Components Required

To implement this without breaking your architecture, you need:

* **`.resx` (Resource) Files:** To hold your translation key-value pairs.
* **`IStringLocalizer<T>`:** The built-in .NET service used to look up strings from your resource files.
* **An Error Translation Service/Shared Resource:** A central spot for exception-related strings so your custom exceptions or middleware can access them.

### Step 1: Create a Shared Resource Class

By default, `IStringLocalizer<T>` looks for resource files that match the namespace of the class `T`. Because exceptions can happen anywhere (Logic, Repository, Middleware), it's best to create a dummy class to act as a **Shared Resource anchor**.

Create an empty class in your Core/Application layer:

```csharp
{
}
```

Next, create your actual resource files in the same folder:

* `ErrorMessages.resx` (Default/Fallback language)

Inside `ErrorMessages.ja.resx`, you might add:

* **Name:** `MachineNotFound` | **Value:** `指定されたマシンが見つかりません。`

### Step 2: Configure Localization in `Program.cs`

You need to tell ASP.NET Core to support localization and look for your resource files. Add this to your backend configuration:

```csharp
var builder = WebApplication.CreateBuilder(args);
```

### Step 3: Update Your Custom Exception Middleware

Since you already have custom Exception classes and Middleware, you can inject `IStringLocalizer<ErrorMessages>` directly into your middleware to translate the error messages before sending the JSON response back to Angular.

Here is how you can clean up the response:

```csharp
using Microsoft.Extensions.Localization;
using System.Net;
using System.Text.Json;

namespace MachineSurveillance.Api.Middleware;

public class ExceptionHandlingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<ExceptionHandlingMiddleware> _logger;

    public ExceptionHandlingMiddleware(RequestDelegate next, ILogger<ExceptionHandlingMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context, IStringLocalizer<ErrorMessages> localizer)
    {
        try
        {
            await _next(context);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "An unhandled exception occurred.");
            await HandleExceptionAsync(context, ex, localizer);
        }
    }

    private static Task HandleExceptionAsync(HttpContext context, Exception exception, IStringLocalizer<ErrorMessages> localizer)
    {
        context.Response.ContentType = "application/json";
        
        var response = new ErrorResponse();
        
        switch (exception)
        {
            case MachineNotFoundException machineEx:
                context.Response.StatusCode = (int)HttpStatusCode.NotFound;
                // Use the localizer to translate the key using the request's culture
                response.Message = localizer["MachineNotFound"]; 
                response.ErrorCode = "MACHINE_NOT_FOUND";
                break;

            // Handle other custom exceptions...

            default:
                context.Response.StatusCode = (int)HttpStatusCode.InternalServerError;
                response.Message = localizer["InternalServerError"];
                response.ErrorCode = "INTERNAL_SERVER_ERROR";
                break;
        }

        var jsonResult = JsonSerializer.Serialize(response);
        return context.Response.WriteAsync(jsonResult);
    }
}

public class ErrorResponse
{
    public string ErrorCode { get; set; } = string.Empty;
    public string Message { get; set; } = string.Empty;
}

```

```csharp
namespace MachineSurveillance.Core.DtoModels;

public class MachineOverviewDto
{
```

```csharp
public async Task<List<MachineOverviewDto>> GetMachineOverviewListAsync()
{
    return await _context.MachineMasters
        .Select(m => new MachineOverviewDto
        {
            MachineId = m.MachineId,
            MachineName = m.MachineName,
            // Subquery to grab ONLY the top 1 latest data row ordered by date
            LatestDataDate = _context.MachineData
                .Where(d => d.MachineId == m.MachineId)
                .OrderByDescending(d => d.DataDate)
                .Select(d => (DateTime?)d.DataDate)
                .FirstOrDefault(),

            LatestData = _context.MachineData
                .Where(d => d.MachineId == m.MachineId)
                .OrderByDescending(d => d.DataDate)
                .Select(d => (double?)d.Data)
                .FirstOrDefault(),

            LatestStatus = _context.MachineData
                .Where(d => d.MachineId == m.MachineId)
                .OrderByDescending(d => d.DataDate)
                .Select(d => d.Status)
                .FirstOrDefault()
        })
        .ToListAsync();
}

```

#### Why this is efficient:

EF Core translates this entire statement into a single, optimized SQL query utilizing `ROW_NUMBER() OVER (...)` or a correlated subquery (depending on your database provider like SQL Server or PostgreSQL). Only one row per machine crosses the network from the DB to your backend application.

### Requirement 2: The Dedicated "MachineData Page" (Full History)

As you mentioned, retrieving the full history for a single machine is straightforward. However, because this table will **populate fast and a lot**, pulling the *entire* history at once will eventually slow down your Angular frontend and cause performance bottlenecks.

Here is how you should structure the Repository method to handle this safely:

```csharp
public async Task<List<MachineDataDto>> GetMachineHistoryAsync(int machineId, int pageNumber = 1, int pageSize = 50)
{
    return await _context.MachineData
        .Where(d => d.MachineId == machineId)
        .OrderByDescending(d => d.DataDate) // Show newest data first
        .Skip((pageNumber - 1) * pageSize)   // Skip past previous pages
        .Take(pageSize)                     // Take only the requested chunk
        .Select(d => new MachineDataDto
        {
            DataDate = d.DataDate,
            Data = d.Data,
            Status = d.Status
        })
        .ToListAsync();
}

```

Since your `MachineData` table uses a Composite Primary Key (`MachineId` + `DataDate`), the database automatically creates a clustered index on these columns.

Because `MachineId` is the first part of that key, queries filtering by `Where(d => d.MachineId == machineId)` and ordering by `DataDate` will be incredibly fast, even with millions of rows. Your schema is perfectly optimized for this type of time-series retrieval!

```csharp
using System;
using System.Text.RegularExpressions;

public void ValidateUserInput(string userId, string password, string username)
{
    // 1. ユーザーIDが未設定の場合 (If User ID is missing/null/empty)
    if (string.IsNullOrWhiteSpace(userId))
    {
        throw new BadRequestException("VALIDATION.USER_ID_REQUIRED"); // 引数:userIdが設定されていません。
    }

    // 2. ユーザーIDが4桁未満の場合 (If User ID is less than 4 characters)
    if (userId.Length < 4)
    {
        throw new BadRequestException("VALIDATION.USER_ID_INVALID_LENGTH"); // 引数:userIdが4桁ではありません。
    }

    // 3. ユーザーIDが半角数字以外が含まれている場合 (If User ID contains anything other than half-width numbers)
    if (!Regex.IsMatch(userId, @"^[0-9]+$"))
    {
        throw new BadRequestException("VALIDATION.USER_ID_NOT_NUMERIC"); // 引数:userIdに半角数字以外が設定されています。
    }

    // 4. パスワードが未設定の場合 (If Password is missing/null/empty)
    if (string.IsNullOrWhiteSpace(password))
    {
        throw new BadRequestException("VALIDATION.PASSWORD_REQUIRED"); // 引数:passwordが設定されていません。
    }

    // 5. パスワードが8文字未満の場合 (If Password is less than 8 characters)
    if (password.Length < 8)
    {
        throw new BadRequestException("VALIDATION.PASSWORD_TOO_SHORT"); // 引数:passwordが8桁以上ではありません。
    }

    // 6. パスワードが少なくとも数字、英文字、記号の組み合わせではない場合 
    // (If Password doesn't contain a mix of numbers, letters, and symbols)
    // Regex breakdown: (?=.*\d) checks for number, (?=.*[a-zA-Z]) checks for letter, (?=.*[^a-zA-Z0-9]) checks for symbol
    if (!Regex.IsMatch(password, @"^(?=.*\d)(?=.*[a-zA-Z])(?=.*[^a-zA-Z0-9]).+$"))
    {
        throw new BadRequestException("VALIDATION.PASSWORD_WEAK_COMPOSITION"); // 引数:passwordが少なくとも数字、英文字、記号の組み合わせが必要です。
    }

    // 7. ユーザー名が未設定の場合 (If Username is missing/null/empty)
    if (string.IsNullOrWhiteSpace(username))
    {
        throw new BadRequestException("VALIDATION.USERNAME_REQUIRED"); // 引数:usernameが設定されていません。
    }

    // 8. ユーザー名が30文字を超える場合 (If Username is more than 30 characters)
    if (username.Length > 30)
    {
        throw new BadRequestException("VALIDATION.USERNAME_TOO_LONG"); // 引数:usernameが30文字以下ではありません。
    }
}

```

It’s completely normal to find the combination of Guards and Interceptors a bit confusing at first since they both deal with security, but they operate at entirely different levels of your Angular application.

If an **AuthGuard** is the security guard standing at the front door deciding who gets into the building, an **HttpInterceptor** is the personal assistant who manages all the paperwork going in and out of the building.

Here is exactly what an interceptor is, what it does for JWT authentication, and how it compares to a guard.

---

## What is an HttpInterceptor?

An `HttpInterceptor` is an Angular feature that allows you to inspect, modify, or block outgoing HTTP requests and incoming HTTP responses globally.

Instead of manually adding your JWT bearer token to every single `httpClient.get()` or `httpClient.post()` call in your services, you write an interceptor **once**. It automatically intercepts every HTTP request right before it leaves the browser and goes to your ASP.NET backend.

---

## What does it do in a JWT Auth Flow?

In a typical Angular application using JWT, the interceptor handles two massive responsibilities: **Attaching the Token** and **Handling Token Expiration (401/403 Errors).**

### 1. Automatically Injecting the JWT (Outgoing)

When you log in, you probably save your JWT token in `localStorage`, `sessionStorage`, or an Auth State. The interceptor grabs that token and clones the outgoing request to add the `Authorization` header.

Here is a simplified look at what an Angular Interceptor looks like in code:

```typescript
import { Injectable } from '@angular/core';
import { HttpInterceptor, HttpRequest, HttpHandler, HttpEvent } from '@angular/common/http';
import { Observable } from 'rxjs';

@Injectable()
export class AuthInterceptor implements HttpInterceptor {
  intercept(req: HttpRequest<any>, next: HttpHandler): Observable<HttpEvent<any>> {
    // 1. Get the token from storage
    const token = localStorage.getItem('jwt_token');

    // 2. If token exists, clone the request and add the Bearer header
    if (token) {
      const cloned = req.clone({
        headers: req.headers.set('Authorization', `Bearer ${token}`)
      });
      return next.handle(cloned); // Pass the modified request forward
    }

    // 3. If no token, just pass the original request through (e.g., login request)
    return next.handle(req);
  }
}

```

### 2. Handling Global API Errors (Incoming)

What happens if the user's JWT expires while they are staring at the page, and they try to click a button? Your ASP.NET backend middleware is going to throw a `401 Unauthorized` response.

Instead of writing error-handling code in every single Angular component, the Interceptor can **catch** that `401` globally, automatically redirect the user back to the login page, or attempt to call your backend's Refresh Token endpoint to get a new JWT seamlessly.

---

## AuthGuard vs. HttpInterceptor: The Ultimate Comparison

To keep them straight in your head, look at how they split up the work:

| Feature | AuthGuard (The Router Guard) | HttpInterceptor (The Network Guard) |
| --- | --- | --- |
| **Where it operates** | Angular Router / UI Navigation | Angular `HttpClient` / Network Layer |
| **Primary Trigger** | Triggered when the user tries to click a link or switch pages/components. | Triggered when your code makes an API call (`GET`, `POST`, etc.) to the backend. |
| **Core Question** | *"Is this user allowed to view this screen?"* | *"Does this network request have the right credentials?"* |
| **Typical Action** | Checks if a token exists; redirects to `/login` if not. | Clones the request, attaches `Bearer <token>`, and handles `401/403` errors from the server. |

### How they work together in a real user flow:

1. The user clicks on the `/dashboard` route.
2. **AuthGuard** kicks in, checks that the user is logged in, and says, *"Yep, you can view this page."*
3. The Dashboard component loads and immediately calls `this.http.get('/api/machine-list')` to fetch data from your ASP.NET backend.
4. **HttpInterceptor** pauses that request, grabs the JWT from storage, sticks it into the header, and fires it off to your API.

They work as a tag-team to make sure your front-end UI stays protected and your back-end API requests stay authenticated!
