# Database structure

In your database, the structure is divided into exactly **two tables** linked by a **One-to-Many relationship**. This prevents data duplication and keeps the structure optimized, even when logging thousands of temperature readings.

Here is the exact schema and data layout for both tables:

---

## 1. The `MachineMaster` Table (The Parent)

This table acts as your master inventory directory. Each machine gets **exactly one row** containing its static configuration details.

* **`MachineID`** is the **Primary Key (PK)**—a unique identifier for each piece of hardware.

### Table Schema & Sample Data:

| MachineID (PK) `[VARCHAR]` | MachineName `[VARCHAR]` | Location `[VARCHAR]` | ModelNumber `[VARCHAR]` |
| --- | --- | --- | --- |
| **M-101** | Conveyor Belt A | Assembly Floor 1 | CV-2000 |
| **M-102** | Main Cooling Pump | Basement Plant | WP-X300 |
| **M-103** | Hydraulic Press | Forging Area | HP-500 |

---

## 2. The `MachineData` Table (The Child)

This table acts as your continuous, time-series telemetry logbook. It stores every single temperature reading from **all** machines sequentially in a single location.

* **`DataID`** is the **Primary Key (PK)** for the log entry itself.
* **`MachineID`** is the **Foreign Key (FK)**. It links the reading back to the parent machine in the master table.

### Table Schema & Sample Data:

| DataID (PK) `[INT]` | MachineID (FK) `[VARCHAR]` | Temperature `[DECIMAL]` | Temperature_Date `[DATETIME]` |
| --- | --- | --- | --- |
| 1 | **M-101** | 42.5 | 2026-05-29 09:00:00 |
| 2 | **M-102** | 18.2 | 2026-05-29 09:00:10 |
| 3 | **M-101** | 43.1 | 2026-05-29 09:01:00 |
| 4 | **M-103** | 78.4 | 2026-05-29 09:01:12 |
| 5 | **M-102** | 18.1 | 2026-05-29 09:02:10 |

## How They Connect (The Entity Relationship)

In Entity Framework Core, your database structure translates to these two clean classes, where EF automatically handles the database constraints:

```csharp
public class MachineMaster
{
    [Key]
    public string MachineID { get; set; } // PK
    public string MachineName { get; set; }
    public string Location { get; set; }
    
    // Entity Framework Navigation Property (One Machine has Many Data logs)
    public ICollection<MachineData> TelemetryLogs { get; set; }
}

public class MachineData
{
    [Key]
    public int DataID { get; set; } // PK
    
    [ForeignKey("MachineMaster")]
    public string MachineID { get; set; } // FK
    
    public double Temperature { get; set; }
    public DateTime Temperature_Date { get; set; }
    
    // Navigation Property back to parent
    public MachineMaster Machine { get; set; }
}

```

### Why this structure works perfectly for your surveillance app:

When your backend runs a query to display a live dashboard line chart for `M-101`, Entity Framework doesn't scan the whole database. It uses the **Foreign Key Index** to instantly pluck out only the rows from `MachineData` where `MachineID == "M-101"`, keeping your application lightning-fast even with millions of entries stored on AWS.

It perfectly matches your goal: the payload hides the real password, the database only stores hashes, and **pasting the payload hash or the database hash into the login field will completely fail.** To make this happen, your backend doesn't *decrypt* the payload back to plain text. Instead, it uses a technique called **Asymmetric Public-Key Encryption** for transit, combined with a **One-Way Hash** for the database.

Here is the exact mechanism of how this works step-by-step.

---

## The Core Concept: Encryption vs. Hashing

To understand the workflow, we have to look at the two different tools being used:

1. **The Transit Shield (Asymmetric Encryption - RSA):** This uses a paired **Public Key** (which can only lock/encrypt data) and a **Private Key** (which lives safely on your ASP.NET backend and can unlock/decrypt data).
2. **The Database Shield (One-Way Hashing - BCrypt):** A scrambled footprint of the password that can never be reversed or decrypted.

---

## Step-by-Step Execution Workflow

### Step 1: Angular Fetches the Public Key

When the user loads the login page, your Angular app makes a quick API call to ASP.NET to ask for the current **Public Key**.

### Step 2: Angular Encrypts the Input (The Payload)

The user types `admin123` into the password box. Right before sending the HTTP POST request, Angular uses the Public Key to **encrypt** the text.

* **The Result:** The payload turns into a massive, unreadable scrambled string (Base64 or Hex).
* **Why pasting this fails:** If a hacker copies this payload string and pastes it into the login box next time, Angular will treat that massive string as the *new plain-text input* and encrypt it *again* using the public key. The backend will decrypt it, see the massive string instead of `admin123`, and reject the login.

### Step 3: ASP.NET Decrypts the Payload

The scrambled payload arrives at your backend. ASP.NET takes its hidden **Private Key** (which never leaves the server) and decrypts the string.

* **The Result:** The backend now temporarily holds the actual plain-text string: `"admin123"`.

### Step 4: ASP.NET Hashes and Compares with the DB

Now that the backend has the real password `"admin123"`, it processes it through its standard hashing function (like BCrypt) and compares it against the secure hash stored in your `UserMaster` database table.

* **Why stealing the DB hash fails:** If a hacker steals your database file and tries to paste the database hash (e.g., `$2a$11$...`) into the front-end login box, Angular will encrypt it, the backend will decrypt it back to `$2a$11$...`, but then the backend will hash that string *again*. A hash of a hash will never match the database record, blocking the attacker completely.

---

## Why this solves your superior's concern

This architecture perfectly fulfills every constraint your team is looking for:

* **Over the network:** The password is completely hidden. Anyone looking at the browser's Network Tab will only see a scrambled cryptographic block, not `admin123`.
* **Inside the Database:** You remain completely compliant with security standards because you are only storing un-reversible BCrypt hashes.
* **Input Integrity:** The system **only** authenticates the request if the human user literally types the characters `a-d-m-i-n-1-2-3` into the Angular UI component.

If you want to recreate this behavior in your project, look up libraries that support **RSA Encryption**.

1. **On Angular:** Use a library like `jsencrypt` or the browser's built-in `window.crypto.subtle` to handle the `encrypt(plainText, publicKey)` step.
2. **On ASP.NET:** Use the built-in `System.Security.Cryptography.RSACryptoServiceProvider` to handle the `Decrypt(payload, privateKey)` step inside your login controller before passing the result to your Entity Framework authentication logic.

## The Architectural Blueprint

```text
1. [Angular] requests Public Key ─────────────────────────> [ASP.NET] returns RSA Public Key
2. [Angular] encrypts "admin123" with Public Key ─────────> [ASP.NET] receives cipher text
3.                                                          [ASP.NET] decrypts cipher with Private Key
4.                                                          [ASP.NET] BCrypt-verifies plain text against DB

```

By passing a newly generated public key or using a session-bound key, the encrypted payload changes every single time. Copying and pasting an old payload or a database string directly into the login field will always result in an authentication failure.

## 1. Frontend Architecture (Angular 21 + TypeScript)

### Implementation Logic

You can use the native browser `window.crypto.subtle` API or a lightweight, proven wrapper library like `jsencrypt`.

**`auth.service.ts` snippet:**

```typescript
import { Injectable, inject } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { JSEncrypt } from 'jsencrypt';
import { firstValueFrom } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class AuthService {
  private http = inject(HttpClient);

  async login(username: string, plainTextPassword: string) {
    // 1. Fetch the public key from ASP.NET
    const { publicKey } = await firstValueFrom(this.http.get<{ publicKey: string }>('/api/auth/public-key'));

    // 2. Encrypt the plain text password
    const encryptor = new JSEncrypt();
    encryptor.setPublicKey(publicKey);
    const encryptedPassword = encryptor.encrypt(plainTextPassword);

    // 3. Send the encrypted string as payload
    return firstValueFrom(this.http.post('/api/auth/login', {
      username,
      password: encryptedPassword // Scrambled RSA string
    }));
  }
}

```

### The Component Implementations

#### A. The Application Layer (`CryptoService.cs`)

This class uses the built-in .NET `RSA` engine to manage keys and handle the decryption phase.

```csharp
using System.Security.Cryptography;

public interface ICryptoService {
    string GetPublicKey();
    string Decrypt(string encryptedBase64Text);
}

public class CryptoService : ICryptoService {
    private readonly RSA _rsa;

    public CryptoService() {
        // Initializes a 2048-bit RSA key pair upon instantiation
        _rsa = RSA.Create(2048);
    }

    public string GetPublicKey() {
        // Export the public key in standard PEM/X509 format for Angular
        return _rsa.ExportRSAPublicKeyPem();
    }

    public string Decrypt(string encryptedBase64Text) {
        byte[] dataToDecrypt = Convert.FromBase64String(encryptedBase64Text);
        byte[] decryptedData = _rsa.Decrypt(dataToDecrypt, RSAEncryptionPadding.OaepSHA256);
        return System.Text.Encoding.UTF8.GetString(decryptedData);
    }
}

```

> **Architecture Note:** Register `ICryptoService` as a **Singleton** in your `Program.cs` (`builder.Services.AddSingleton<ICryptoService, CryptoService>();`) so the private key stays active in application memory across requests.

#### B. The Service Layer (`UserService.cs`)

Once the text is decrypted, the service uses `BCrypt` to match the value against your Entity Framework data models.

```csharp
public class UserService {
    private readonly AppDbContext _context;
    private readonly ICryptoService _cryptoService;

    public UserService(AppDbContext context, ICryptoService cryptoService) {
        _context = context;
        _cryptoService = cryptoService;
    }

    public bool ValidateUser(string username, string encryptedPasswordPayload) {
        // 1. Decrypt RSA back to plain text ("admin123")
        string plainTextPassword = _cryptoService.Decrypt(encryptedPasswordPayload);

        // 2. Fetch user entity via EF Core
        var user = _context.Users.FirstOrDefault(u => u.Username == username);
        if (user == null) return false;

        // 3. Verify plain text against the slow database BCrypt hash
        return BCrypt.Net.BCrypt.Verify(plainTextPassword, user.PasswordHash);
    }
}

```

#### C. The API Layer (`AuthController.cs`)

Exposes the two needed access points to your Angular frontend.

```csharp
[ApiController]
[Route("api/[controller]")]
public class AuthController : ControllerBase {
    private readonly ICryptoService _cryptoService;
    private readonly UserService _userService;

    public AuthController(ICryptoService cryptoService, UserService userService) {
        _cryptoService = cryptoService;
        _userService = userService;
    }

    [HttpGet("public-key")]
    public IActionResult GetPublicKey() {
        var publicKeyPem = _cryptoService.GetPublicKey();
        return Ok(new { publicKey = publicKeyPem });
    }

    [HttpPost("login")]
    public IActionResult Login([FromBody] LoginDto dto) {
        bool isSuccess = _userService.ValidateUser(dto.Username, dto.Password);
        if (!isSuccess) return Unauthorized("Invalid credentials");
        
        // Issue JWT token or establish session here...
        return Ok(new { message = "Logged in successfully" });
    }
}

```
